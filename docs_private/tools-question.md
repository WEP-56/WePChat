1. edit_file 过于严格
- 改动建议：
匹配失败时返回提示：“未找到匹配内容，当前文件前 200 字符为：...” 
可选支持**正则模式**（通过参数 `useRegex: true`）
或支持忽略空白差异的模式（`ignoreWhitespace: true`）
- 为何这么改：
find 必须**完全一致**，包括空格、换行、引号风格（单 vs 双），否则静默不生效
没有匹配失败的错误提示，工具返回成功但什么都没改，容易让LLM困惑

2. `preview_html` vs `write_file` — 两种模式的流程割裂
- 改动建议
直接删除 preview_html 因为这是旧资产，同时强化write_file，增加可选参数 `preview: true`，模型可以选择写入的哪个html可以自动打开预览
- 为什么这么改
preview_html 是早期开发时，无工作区概念的旧资产。已经不需要，同时二者共存存在割裂，弱模型会难以取舍使用哪个工具。应当把preview_html的部分功能移入write_file，并彻底删除preview_html

3. `run_js` — 沙盒与工作区隔离，无法读取文件
- 改动建议 
提供一个轻量的 `SandboxFS` 接口：`readFile(path)` 和 `writeFile(path, content)`，仅在沙盒内可用，限制在当前会话工作区 
或者允许在调用 `run_js` 时通过参数传入工作区文件内容（例如 `inputFiles: { "data.json": "./data.json" }`）
- 为什么这么改
沙盒只能处理手动注入的字符串数据，无法直接读写工作区，想对文件内容做转换 → 得先 `read_file` 拿到内容 → 手动粘贴进 `run_js` 代码 → 处理完 → 再用 `write_file` 写回去，流程较长

5. `image_go` — size 参数支持不明确
- 改动建议：
在工具描述里写明支持的尺寸列表（例如 `1024x1024`, `1024x1536`, `1536x1024`, `512x512` 等）
- 为什么这么改：
这是来自LLM的反馈，它说“生成鲸图片时，文档说 size "未知用 auto 或留空"，但传 `auto` 时不确定底层 API 是否接受，最后还是猜了一个常用的 `1024x1024`。”

6. 工具链的“数据管道”能力缺失
**具体问题**：
- 工具之间是**孤立的**，输出不能作为下一个工具的输入自动传递
- 复杂流程需要多次交互，每次LLM都得说“上面那步的结果是…，现在继续…”
**改进建议**：
最简单的方案：工具调用的返回值可以在后续调用中以 `{{prev.result}}` 或 `$1` 的形式引用
更通用的方案：支持定义**多步骤工作流**（类似 GitHub Actions 的 step），让工具链能自动串联

7. `delete_file` — 每次需要用户确认，批量清理不便
- 改动建议：
为delete_file添加批量删除能力，但删除确认仍然在用户手里，区别是删除确认弹窗可以显示更多文件
- 为什么这么改：
每删一个文件都要弹用户确认框，连续删多个时交互成本高
没有“删除整个目录”或“批量删除”的能力

8. `read_file` — 不支持读取片段
- 改动建议：
增加可选参数例如
```
read_file(path, lines: "1-20")   // 第 1 到 20 行
read_file(path, lines: "50-80")  // 第 50 到 80 行
read_file(path, lines: "1-")     // 从第 1 行开始
read_file(path, lines: "-30")    // 最后 30 行
```
- 为什么这么改：
想只看前 20 行或第 50~80 行，做不到
大文件时没法做“预览式读取”

9. `list_files` — 不显示目录结构，只能平铺
- 改动建议：
或者返回结构化数据，区分文件 vs 文件夹，并显示层级关系
- why
不知道文件夹中有哪些子文件夹
想查看某个文件夹的内容？不行，`list_files` 没有 `path` 参数
对于多级目录的项目，只能靠猜测文件夹路径，然后用 `read_file` 试探

10. 文件夹创建/删除 — 不存在对应工具
**现状**：

| 操作 | 是否支持 | 说明 |
|:----|:--------:|:----|
| 创建文件夹 | ❌ | `write_file("folder/file.txt")` 可以**隐式**创建文件夹，但无明确 `mkdir` 工具 |
| 删除文件夹 | ❌ | 只能用 `delete_file` 逐个删文件，文件夹本身无法删除 |
| 重命名/移动 | ❌ | 无相关操作 |

**问题**：
- 想建一个空文件夹？做不到
- 想清空整个项目目录？只能一个一个删文件，效率极低
- 想确认文件夹是否存在？只能 `list_files` 后自己 parse 路径

**建议**：
- 增加 `create_folder(path)` 工具（明确创建空目录）
- `delete_file` 支持目录删除（级联删除内部所有文件）
- 或至少让 `list_files` 能列出文件夹，让 LLM 知道目录结构
